# Branch Context

## Что важно знать про ветки

Сейчас отчёт нельзя строить по старым feature-веткам как по актуальному состоянию системы.

## Почему

- `origin/feature/market-data` уже не имеет уникальных коммитов относительно `origin/main`
- `origin/feature/mobile-backend-db` тоже не имеет уникальных коммитов относительно `origin/main`
- актуальная интеграция модулей находится в `main`

## Особый случай фронтенда

Android-клиент пришёл не из ветки этого репозитория, а из внешнего репозитория `MaryShust/Stock-Exchange`.

Это значит, что история проекта состоит из двух линий:

- внутренняя Git-история backend и инфраструктуры
- внешний импорт Android-прототипа

## Практический вывод

В отчёте нужно честно писать, что:

- основной baseline задаёт `main`
- старые feature-ветки помогают понять этапы развития backend
- Android-клиент пока находится на стадии интеграции в общий контур
