# Тестирование проекта

## Что требует курс

По материалам курса в отчёте нужно раскрывать три уровня тестирования:

- модульное;
- интеграционное;
- системное.

## Как это переносится на наш проект

### Модульное тестирование

Проверяет отдельные классы, use case и вспомогательную логику:

- регистрация пользователя;
- покупка акций и проверки денежного баланса;
- логика продажи;
- расчёт статистики;
- парсинг котировок.

### Интеграционное тестирование

Проверяет взаимодействие нескольких частей системы:

- HTTP routes + use case;
- репозитории + тестовая БД;
- gateway + upstream;
- data layer Android-клиента + mock server.

### Системное тестирование

Проверяет сквозной пользовательский сценарий от клиента до backend и observability stack.

## Где используется у нас

- backend tests: `mobile-backend/src/test`
- quotes-service tests: `quotes-service/internal/app/http_test.go`, `quotes-service/internal/quotes/parser_test.go`, `quotes-service/internal/quotes/store_test.go`
- Android test scaffolding: `android-app/app/src/test`, `android-app/app/src/androidTest`
- load-test result for public quote endpoint: зафиксирован в отчётных материалах проекта

## Текущее состояние проекта

- `mobile-backend` покрыт лучше всего;
- у backend уже есть отдельный тест на покупку и проверки денежного баланса;
- `quotes-service` уже имеет полезные unit/integration тесты;
- `mobile-api` пока почти без тестов;
- `android-app` пока имеет только стартовый каркас для тестирования.

## Что изменилось в последней версии backend

После обновления `main` в серверной части появились:

- отдельный сценарий пополнения денежного баланса;
- проверки достаточности средств перед покупкой;
- более корректная обработка запроса владения акцией, когда символ в портфеле отсутствует.

## Что важно для защиты

Нужно уметь объяснить, что тестирование в проекте уже есть, но его зрелость неравномерна. Это естественно для системы, которая собиралась поэтапно и из нескольких модулей.

## Что логично сделать дальше

- добавить тесты на `mobile-api`;
- подготовить mock-сервер для Android data layer;
- собрать хотя бы один end-to-end smoke-сценарий;
- связать системные тесты с проверкой traces.
